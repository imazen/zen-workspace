# Recovery plan: forgotten ground across the zen-* repos

**Scope:** zensim, zenanalyze (zentrain + zenpredict + zenpicker), zenmetrics, zenavif, zenwebp, coefficient. Auxiliary: zenjpeg, zenjxl variants where they overlap.

**Goal:** recover proven-out research into mainline; merge true improvements; refuse abandoned bloat. Then — and only then — finalize ZNPR v3 + minimize zenpredict's API.

**Hard constraints**
- Do NOT finalize ZNPR v3 / minimize zenpredict until Phase 3 confirms no further requirements surface from the recovered branches.
- zenavif & zenwebp 0.x releases were yanked because they shipped baked predictors; future re-releases must consume a **stable** zenpredict format.
- Anti-bloat rule (per `CLAUDE.md`): every cherry-pick needs documented improvement (≥0.005 SROCC or analogous), held-out validation, no new deps unless justified. Code without measured improvement → archive on a `recovered-archive/<topic>` branch, **not** main.
- Per global rule: never touch other repos' uncommitted work. Each repo's recovery happens in its own working tree.

## Phase 1 — Read & distill (next session block)

**Goal:** build a per-branch register of every claimed result, with verdict.

**Per branch with last commit ≥ 2026-04-25:**

1. Read every `*.md` file (tracked AND gitignored), every `benchmarks/**/*.{md,log,csv,tsv}` summary, every `docs/**`. Use `find . -type f \( -name '*.md' -o -name '*.log' \) -newer .git -not -path '*/target/*' -not -path '*/node_modules/*'`.
2. For each finding, capture in `RECOVERY_REGISTER.md` (per-repo file):
   - **What was tried** (hypothesis + config)
   - **What was measured** (numbers, dataset, n)
   - **Verdict**: kept / abandoned / superseded / unverified

   - **Cross-refs** to any duplicate/related work on sibling branches
3. Surface the **dormant** sibling worktree dirs: `git reflog --all` per dir; if nothing relevant since 2026-04-01, schedule `git worktree remove` after recovery is complete.

**Output artifact** (one per repo): `<repo>/docs/RECOVERY_REGISTER_2026-05-08.md` listing every experiment with verdict.

**Time estimate:** ~6-10 hours of focused reading. Can be split across sessions; the register is incremental.

## Anti-bloat rules

For each cherry-pick:
1. Documented numerical improvement OR clear API simplification OR security/correctness fix.
2. No new public API surface unless explicitly requested.
3. No new dependencies unless justified in the commit message.
4. Code without a measured improvement → archive on `recovered-archive/<topic>` branches; **do not** merge to main.
5. Each cherry-pick is squashed into a single commit with the register's verdict in its message.

## Order & gates

- Phase 0: **DONE** (this doc).

- Phase 2: per-repo PRs may be parallelized once registers are settled.

- Phase 4 blocked by Phase 3 acceptance.

## Decisions (confirmed 2026-05-08)

1. **ZNPR v3 is canonical for everyone.** zensim, zenavif, zenwebp, zenpicker, every consumer re-bakes to v3 once Phase 4 ships v3. v2 remains read-only for prior bakes.
2. **Caller-supplied bakes only.** zenavif/zenwebp re-releases will NOT bundle a bake. Callers (imageflow, picker, etc.) supply bytes. Removes the lifecycle coupling that caused the yank.
3. **Dormant zensim sibling worktrees → `archive/`.** Anything pre-2026-04-25 with no unmerged value after Phase 1 read goes to `archive/<branchname>` namespace, then `git worktree remove`. No silent deletion.
4. **zentrain tool location:** `zenanalyze/zentrain/tools/zensim_metric_train.py` (matches `train_hybrid.py` convention). Config example at `zenanalyze/zentrain/examples/zensim_metric_config.py` (matches per-codec picker examples shape). Reuses the existing pipeline → inherits inspection / safety / triage / ablation tooling for free.

## Constraint: no crate publishes during this recovery cycle

All Phase 2-4 work lands as commits on main. Crate version bumps stay on the unreleased changelog section. **No `cargo publish` until the user explicitly approves a release window** after Phase 4 acceptance.
